# Fix: Index Page Timer Does Not Navigate

## Goal

Make the `Index` page automatically navigate to the `Main` page after the 1-second `setTimeout` in `aboutToAppear`.

## Scope / Non-goals

- **In scope**: Fix the navigation mechanism so `pushPathByName("MainPage", ...)` actually triggers a page transition.
- **Not in scope**: Any UI changes to Index or Main pages beyond what's needed for navigation to work.

## Current State And Constraints

1. **`Index.ets`** declares `NavPathStack` and calls `pushPathByName("MainPage", null)` in `aboutToAppear` after 1 second.
2. **`Index.ets` `build()`** renders only a `RelativeContainer` â€” **no `Navigation` component**.
3. **`router_map.json`** correctly registers `"MainPage"` â†’ `src/main/ets/pages/Main.ets` â†’ builder `load`.
4. **`Main.ets`** has both `@Entry` + `@Component` on `struct Main`, **and** an exported `@Builder function load()`. When used as a NavDestination, `@Entry` is incorrect â€” it marks the struct as an independent page entry, which conflicts with being a NavDestination.
5. **`main_pages.json`** only lists `pages/Index` â€” this is correct for the entry page.

### ArkTS-specific constraints

- Navigation with `NavPathStack` requires a `Navigation` component wrapping the content.
- `router_map.json` + `navDestination` builder is the recommended approach.
- NavDestination components should NOT have `@Entry` â€” they are not independent pages.

## Design

### Root cause

`NavPathStack.pushPathByName()` is a **data operation** on the stack. Without a `Navigation` component that uses this stack as its controller, the push has no visual effect. The `Navigation` component listens to stack changes and renders the corresponding `NavDestination` based on `router_map.json`.

### Solution

**Step A** â€” Wrap `Index.ets` content in `Navigation(this.NavPathStack)`.

```arkts
build() {
  Navigation(this.NavPathStack) {
    RelativeContainer() {
      // ... existing content unchanged
    }
    .width('100%')
    .height('100%')
  }
  .width('100%')
  .height('100%')
  .hideTitleBar(true)
  .hideToolBar(true)
}
```

This allows the `Navigation` component to intercept `pushPathByName` calls and resolve them against `router_map.json`.

**Step B** â€” Fix `Main.ets` to remove `@Entry` (since it is now a NavDestination, not an independent page). The `@Builder` builder pattern is correct for NavDestination.

```arkts
// Remove @Entry â€” NavDestination pages should not be independent entries
@Component
struct Main {
  build() {
    // ... existing content
  }
}
```

The `@Builder export function load()` at the bottom stays â€” this is the entry point used by `router_map.json`.

## Risks And Compatibility

- **Navigation chrome**: Adding `Navigation` may introduce a title bar or toolbar. The `.hideTitleBar(true).hideToolBar(true)` mitigates this for the splash screen.
- **`@Entry` removal on `Main.ets`**: If `Main.ets` is also referenced as a standalone entry page elsewhere (unlikely given current config), removing `@Entry` would break that use case. A grep for `Main` in `main_pages.json` and other configs confirms it is only used via `router_map.json` â€” safe to remove.
- **`router_map.json` compatibility**: Requires API 12+ (HarmonyOS Next). The project already has this configured, so it's the intended approach.

## Rollback

1. **Revert `Index.ets`**: Remove the `Navigation(...)` wrapper, restore `RelativeContainer` as the direct root.
2. **Revert `Main.ets`**: Add back `@Entry` before `@Component`.
3. Run build to confirm rollback is clean.
